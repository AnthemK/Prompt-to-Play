# Lite TRPG Sim Doc Governance

> 适用场景：当一个项目同时有玩家、故事作者、系统开发者、多 agent 协作者时，文档很容易“什么都写一点”，最后所有文档都变得难用。

## 1. 先按读者分文档，不要按“我想到什么”写

- `README.md`
  - 只给玩家和普通使用者看
  - 只回答：
    - 这是什么游戏
    - 有什么内容
    - 核心规则是什么
    - 怎么启动和游玩
    - 如果想深入，去哪里看
- `MEMORY.md`
  - 只给当前开发迭代和后续接手 agent 看
  - 记录：
    - 当前目标
    - 本阶段进度
    - 已完成事项
    - 已知问题
    - 目录快照
    - 文档边界约定
- `TODO.md`
  - 只谈迭代路线和优先级
  - 不承担规则书功能，不承担项目介绍功能
- `docs/*.md`
  - 只放技术细节
  - 每份文档都应写清“面向谁，不面向谁”
- `backend/stories/README.md`
  - 只给故事作者看
  - 不应混入玩家说明或系统开发流水账

## 2. 根 README 的常见污染源

- 写太多架构细节
- 写太多 CLI / 测试 / 校验命令
- 把开发者心里的“系统路线判断”塞给玩家
- 把 API、状态结构、结算字段直接放进玩家说明

推荐做法：

- 根 README 只保留“玩家真的会用到的信息”
- 技术材料统一变成：
  - 简短入口
  - 指向专门文档的链接

## 3. 给每份文档补一句“受众声明”

非常有效，成本也低。

Example:

- Audience: developers maintaining system boundaries and story integration.
- Not for players; player entry stays in the root `README.md`.

作用：

- 后续 agent 更不容易把内容写串
- review 时更容易判断“这段话放错地方了吗”

## 4. 文档索引要单独做

当项目开始出现多份技术文档后，建议新增一个索引文件，例如：

- `docs/TECHNICAL_OVERVIEW.md`

这个文件不讲实现细节，专门讲：

- 每份文档负责什么
- 什么时候该看哪一份
- 哪些文档不能混用

## 5. 多 agent 协作下的文档维护建议

- 每次完成一类工作，就顺手同步对应文档
- 不要等“最后统一补文档”
- 改系统接口时，至少检查：
  - 玩家 README 是否受影响
  - 故事作者文档是否受影响
  - 技术文档是否受影响
  - MEMORY 是否需要更新现状
  - TODO 是否需要调整路线

## 6. 最小执行清单

- 根 README 是否还是玩家向
- TODO 是否只谈路线
- MEMORY 是否只谈开发记忆
- `docs/TECHNICAL_OVERVIEW.md` 是否存在并可导航
- 技术文档是否都写明受众
- 故事作者文档是否独立存在

## 7. 这次治理的直接经验

- 当 README 同时服务“玩家 + 开发者 + 架构设计 + 校验工具说明”时，它一定会失焦
- 最好的处理方式不是“稍微删一点”，而是重新定义文档边界，再整体搬迁内容
- 一份清晰的文档体系，本质上也是代码库治理的一部分

## 8. 技术文档之间也要避免互相抢话题

常见问题：

- `ARCHITECTURE.md` 开始解释一大堆字段结构
- `RUNTIME_STATE.md` 又开始讨论模块职责和功能应该落在哪一层

更好的拆法：

- `ARCHITECTURE.md`
  - 只讲边界、分层、职责、数据流、功能应该放在哪里
- `RUNTIME_STATE.md`
  - 只讲 `state / view / save_data / resolution` 的结构和字段语义

判断方法：

- 如果问题是“这段逻辑该写在哪？”看架构文档
- 如果问题是“这个字段现在长什么样？”看运行态文档

## 9. 阶段文档和长期规范文档也要拆开

另一个常见混淆：

- `ENGINEERING.md` 写进了很多“本阶段已经做完什么”
- `STAGE_DELIVERY.md` 又顺手承担了长期工程规范

推荐拆法：

- `ENGINEERING.md`
  - 永久性的工程纪律
  - 注释标准、命名标准、review 标准、文档同步纪律
- `STAGE_DELIVERY.md`
  - 当前阶段的交付快照
  - 验收命令、Demo 覆盖、已知非阻塞边界、下一阶段起点

判断方法：

- 如果这条内容换到下一个里程碑依然成立，它更像工程规范
- 如果这条内容只描述“当前阶段做到哪里”，它更像交付快照

## 10. TODO 不该是功能愿望池

一个健康的 `TODO.md` 更像“路线判断文档”，而不是“想到什么就记什么”。

建议结构：

- 先写产品方向过滤器
- 再写当前里程碑
- 再写 3 到 4 个当前优先块
- 再写明确延后的事项
- 最后写里程碑完成条件

这样做的好处：

- 多 agent 更容易判断什么是当前主线
- review 时更容易发现路线漂移
- 项目不容易被“大而全冲动”带偏

## 11. 玩家 README 的信息顺序很重要

如果根 `README.md` 已经确认是玩家文档，它最好按下面顺序组织：

1. 这是什么
2. 怎么最快启动
3. 现在能玩什么
4. 一局会经历什么
5. 规则速览
6. 存档和常见问题
7. 想深入时再去哪里看

这样做的原因：

- 玩家最先需要的是“能不能马上玩”
- 然后才是“这个游戏大概玩什么”
- 最后才是“如果我想折腾项目本身，还能去哪看”

像图标、自定义、技术索引这类内容，应该尽量下沉到文末或专门文档。

## 12. 技术文档索引最好按“任务”导航，不只按“文件名”导航

很多技术索引的问题是：

- 列了所有文档
- 但没告诉接手者“我现在遇到这个问题应该先看哪一份”

更有效的做法：

- 给出 5 分钟接手顺序
- 给出按任务分类的阅读路径
- 给出“改动 X 应该更新哪份文档”的提示
这样一个新 agent 才能尽快进入有效工作，而不是先在文档之间来回跳。

## 13. 作者指南应先教“如何开始”，再讲“完整语义”

对于像 story authoring 这种文档，最常见的问题是：

- 一上来就展开大量字段说明
- 作者第一次写故事时，不知道第一步该干什么

更好的顺序：

1. Fast start
2. Safe workflow
3. What belongs here vs in the engine
4. Lite writing principles
5. Common mistakes
6. Checklist
7. 再把字段细节交给 reference 文档

这样“作者指南”和“接口速查表”才不会彼此重复。
