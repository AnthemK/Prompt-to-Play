# Creative Playbook（跨项目创作经验手册）

> 目标：给多个 agent 一套能直接执行的共识，而不是抽象口号。

## 1. 项目架构与管理

### 1.1 先定边界，再写功能

- 先回答三个问题：
  - 前端负责什么
  - 后端负责什么
  - 内容（故事/资源）放哪里
- 内容尽量数据化（JSON/YAML），避免写死在 UI 或系统层。

### 1.2 先保可运行，再追求“完美”

- 每次迭代都保持“可启动、可试玩、可回滚”。
- 大改前先跑一遍最小 smoke test，改完再跑一次。

### 1.3 多 agent 协作最小交接包

- 当前目标（1 句话）
- 已完成内容（列表）
- 正在做什么（列表）
- 下一步（最多 3 条）
- 风险/假设（列表）

## 2. 代码实现经验

### 2.1 可扩展系统的核心原则

- 系统逻辑与题材逻辑解耦
- 统一接口优先于“到处特判”
- 结果结构可解释（谁触发、为何触发、影响了什么）

### 2.2 低技术债习惯

- 修改代码时同步更新：
  - 注释 / docstring
  - README / Memory / 技术文档
- 不留空函数与伪实现
- 命名清晰区分：`template / runtime / state / view`

### 2.3 评审友好写法

- 函数短小、单一职责
- 关键分支前给 1 行注释说明“为什么”
- 数据结构字段尽量稳定，新增字段优先向后兼容

### 2.4 文档也要分层

- 当一个主题既需要“上手指南”又需要“字段速查”时，优先拆成两份文档
- 指南文档负责：
  - 怎么开始
  - 推荐顺序
  - 常见坑
  - 实操建议
- 速查文档负责：
  - 字段名
  - 接口形状
  - 支持的枚举值
- 这样比把两者揉在一起更利于后续多 agent 维护

### 2.5 Git 卫生要分两步做

- 只修改 `.gitignore` 还不够，它只能阻止“以后继续跟踪”
- 如果临时文件已经被 Git 跟踪过，还需要执行：
  - `git rm --cached`
- 这一步不会删除本地文件，但会把它们从版本控制里移出去
- 常见需要处理的垃圾文件：
  - `.DS_Store`
  - `.pycache/` / `__pycache__/` / `*.pyc`
  - 本地日志、临时文件、编辑器交换文件
- 在多项目仓库里，顶层 `.gitignore` 最好统一覆盖这些通用噪音

## 3. 文本内容创作经验（剧情/设定/文案）

### 3.1 可玩优先

- 每段文本都要服务决策或反馈
- 长段叙事要拆成可读块，避免“屏幕墙”
- 关键抉择必须有明确代价与后果

### 3.2 跑团感提升法

- 给同一问题 2-3 种解决方式（战斗/潜行/交涉/调查）
- 失败不要只“卡死”，要推进到“带代价的下一状态”
- 用日志或结算面板解释检定过程

## 4. 图片创作经验（UI/素材）

- 先统一视觉规则：主色、字体、对比度、留白
- 优先可读性：文本区域永远先于装饰
- 资源命名要语义化，避免 `final2/final3`

## 5. 音频创作经验（BGM/音效）

- 先定义音频分层：环境底噪、事件提示、UI 反馈
- 保持动态范围，不要让 BGM 压过对白文本阅读
- 给关键节点预留静音或弱化段，突出抉择紧张感

## 6. 通用检查清单（每次阶段收尾）

- 能否一键启动
- 关键流程是否可走通
- 主要机制是否有最小测试
- 文档是否与实现一致
- 是否留下阻碍下个 agent 的隐性上下文
