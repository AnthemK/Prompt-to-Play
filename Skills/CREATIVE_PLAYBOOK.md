# Creative Playbook（跨项目创作经验手册）

> 目标：给多个 agent 一套能直接执行的共识，而不是抽象口号。

## 1. 项目架构与管理

### 1.1 先定边界，再写功能

- 先回答三个问题：
  - 前端负责什么
  - 后端负责什么
  - 内容（故事/资源）放哪里
- 内容尽量数据化（JSON/YAML），避免写死在 UI 或系统层。

### 1.2 先保可运行，再追求“完美”

- 每次迭代都保持“可启动、可试玩、可回滚”。
- 大改前先跑一遍最小 smoke test，改完再跑一次。

### 1.3 多 agent 协作最小交接包

- 当前目标（1 句话）
- 已完成内容（列表）
- 正在做什么（列表）
- 下一步（最多 3 条）
- 风险/假设（列表）

## 2. 代码实现经验

### 2.1 可扩展系统的核心原则

- 系统逻辑与题材逻辑解耦
- 统一接口优先于“到处特判”
- 结果结构可解释（谁触发、为何触发、影响了什么）

### 2.2 低技术债习惯

- 修改代码时同步更新：
  - 注释 / docstring
  - README / Memory / 技术文档
- 不留空函数与伪实现
- 命名清晰区分：`template / runtime / state / view`

### 2.3 评审友好写法

- 函数短小、单一职责
- 关键分支前给 1 行注释说明“为什么”
- 数据结构字段尽量稳定，新增字段优先向后兼容

### 2.4 给 lite 系统加新机制的推荐方式

- 先扩展“统一合同”，再扩展具体故事
  - 例如先定义：
    - 顶层元数据
    - runtime state 字段
    - view / resolution 字段
    - validator 规则
  - 再让某个 Demo 或样例故事接入
- 新机制优先做成“可选层”
  - 不要强迫旧内容立刻重写
  - 例如：
    - 属性之上增加可选技能层
    - 状态之上增加可选持续时间
- 第一版只解决一个核心表达问题
  - 不要把“技能”一步做成专长树
  - 不要把“状态持续时间”一步做成复杂 buff 引擎
- 先让系统和故事保持清楚分工：
  - 系统负责解释规则
  - 故事负责提供词汇、数值和内容语义

常见坑：

- 直接在某个故事里硬写规则特判
- 前端先长出故事逻辑，后端再补合同
- 为了“可扩展”一次塞进太多未被使用的字段

### 2.5 文档也要分层

- 当一个主题既需要“上手指南”又需要“字段速查”时，优先拆成两份文档
- 指南文档负责：
  - 怎么开始
  - 推荐顺序
  - 常见坑
  - 实操建议
- 速查文档负责：
  - 字段名
  - 接口形状
  - 支持的枚举值
- 这样比把两者揉在一起更利于后续多 agent 维护

### 2.6 Git 卫生要分两步做

- 只修改 `.gitignore` 还不够，它只能阻止“以后继续跟踪”
- 如果临时文件已经被 Git 跟踪过，还需要执行：
  - `git rm --cached`
- 这一步不会删除本地文件，但会把它们从版本控制里移出去
- 常见需要处理的垃圾文件：
  - `.DS_Store`
  - `.pycache/` / `__pycache__/` / `*.pyc`
  - 本地日志、临时文件、编辑器交换文件
- 在多项目仓库里，顶层 `.gitignore` 最好统一覆盖这些通用噪音

### 2.7 发布收尾时，先确认“语义一致”，再看测试是否通过

- 测试全绿不代表玩法语义已经对齐
- 尤其在 story-driven 系统里，最常见的问题不是代码崩掉，而是：
  - 条件过宽
  - 出口并存但语义冲突
  - 某个状态已经成立，但玩家没有收到明确反馈
- 推荐顺序：
  - 先看真实存档 / debug trace
  - 再确认当前故事定义和引擎默认规则是否真的一致
  - 最后才决定改系统层还是改故事层

常见判断规则：

- 如果多个故事都会受益，优先改系统层
- 如果只是某个故事的出口语义太宽，优先改故事包配置
- 如果状态变化已经发生，但玩家不知道，优先补“明确反馈”而不是直接自动结局

### 2.8 Demo 不只是“能测到”，还要“测得清楚”

- 一个好的 Demo 应该把核心机制做成玩家能明确感知的段落
- 不要只让机制埋在长路线里“顺便出现”
- 对于新加入的系统能力，优先考虑：
  - 是否需要一个显式演示节点
  - 是否需要一条单独验收路线

推荐做法：

- 新机制进入系统后，按这个顺序接入：
  - 先让底层合同和测试通过
  - 再让 Demo 有一个最小但明确的展示段
  - 再补自动验收路线
  - 最后补 walkthrough / README

典型例子：

- `attribute + skill`
  - 不能只隐含在已有行动里
  - 最好给 Demo 一个短小的技能试探段
- 遭遇出口解锁
  - 不能只让新动作静默出现
  - 应该明确告诉玩家“现在出现了新的收尾窗口”

## 3. 文本内容创作经验（剧情/设定/文案）

### 3.1 可玩优先

- 每段文本都要服务决策或反馈
- 长段叙事要拆成可读块，避免“屏幕墙”
- 关键抉择必须有明确代价与后果

### 3.2 跑团感提升法

- 给同一问题 2-3 种解决方式（战斗/潜行/交涉/调查）
- 失败不要只“卡死”，要推进到“带代价的下一状态”
- 用日志或结算面板解释检定过程

## 4. 图片创作经验（UI/素材）

- 先统一视觉规则：主色、字体、对比度、留白
- 优先可读性：文本区域永远先于装饰
- 资源命名要语义化，避免 `final2/final3`

## 5. 音频创作经验（BGM/音效）

- 先定义音频分层：环境底噪、事件提示、UI 反馈
- 保持动态范围，不要让 BGM 压过对白文本阅读
- 给关键节点预留静音或弱化段，突出抉择紧张感

## 6. 通用检查清单（每次阶段收尾）

- 能否一键启动
- 关键流程是否可走通
- 主要机制是否有最小测试
- 文档是否与实现一致
- 是否留下阻碍下个 agent 的隐性上下文
- 如果要封版：
  - 先冻结版本范围
  - 只允许 bug 修复、验收修复、文档收口
  - 不要边做 release closure，边偷偷开启下一个里程碑
