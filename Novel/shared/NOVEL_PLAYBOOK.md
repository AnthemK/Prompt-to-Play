# Novel Playbook

> 目标：给多个小说项目和多个协作者一套可执行的共识，而不是抽象口号。

## 1. 项目治理

### 1.1 一书一目录

- 每个小说项目独立管理
- 不把不同书的设定、提纲、正文混在一起
- 跨项目经验统一沉淀到 `shared/`

### 1.2 先定文档边界，再开始堆内容

- `README`
  - 项目入口与导航
- `MEMORY`
  - 当前状态与交接
- `TODO`
  - 下一阶段计划
- `docs`
  - 长期参考资料
- `01-planning` 到 `05-release`
  - 生产过程材料

### 1.3 先把“这本书卖什么”写清，再扩设定

开工前先回答：

- 核心题材是什么
- 读者期待的爽点是什么
- 主角的基本驱动力是什么
- 长线矛盾靠什么持续

如果这四个问题没写清，世界观越大越容易失焦。

## 2. 设定与提纲

### 2.1 设定服务剧情，不服务空转百科

- 只写会进入剧情的规则
- 只细化会影响人物选择的设定
- 背景资料如果长期不进入剧情，就先降级为备注

### 2.2 先搭主干，再补枝叶

推荐顺序：

1. 作品定位
2. 主角弧线
3. 核心冲突
4. 故事总纲
5. 卷纲
6. 章纲
7. 补世界观细节

### 2.3 角色设定要写“功能”

角色卡不只写性格和经历，还要写：

- 他在主线里的作用
- 他和主角的关系变化
- 他代表哪类冲突或诱惑
- 他在哪些阶段必须出场

## 3. 正文生产

### 3.1 草稿优先保推进感

- 每章都要有明确推进
- 推进可以是事件、关系、信息或状态变化
- 不能连续几章只有气氛，没有局势变化

### 3.2 一章只承担有限目标

写章前先写清本章要完成什么：

- 推进哪条冲突
- 释放哪条信息
- 塑造谁
- 留下什么钩子

### 3.3 审校时先抓结构问题

优先级建议：

1. 节奏是否拖沓
2. 角色行为是否自洽
3. 信息是否前后冲突
4. 语言是否顺

不要一上来就只修辞句。

## 4. 跨项目经验沉淀

### 4.1 阶段经验先写进更新日志

- 新经验先记到 `STAGE_UPDATES.md`
- 等规则稳定后，再整理进本文件

### 4.2 经验要可执行

每条经验尽量包含：

- 适用场景
- 推荐做法
- 常见坑
- 最小检查点
