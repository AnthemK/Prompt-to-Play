# Workflow

## Audience

面向维护《观化》生产流程的任何协作者。

不面向读者，也不承担具体设定或剧情内容。

## 正式层与运行层

生产流程默认围绕正式层展开。

正式层承担长期结果：

- 规则
- 设定
- 角色
- 大纲
- 正文
- 修订

运行层只承担临时工作支持：

- `08-experimental-writing/`
- `09-cache/`

运行层中的内容，若形成稳定成果，必须回收到正式层；不能让运行层长期替代正式文档。

## 为什么不是简单的“设定 -> 大纲 -> 正文”

这个粗流程有两个常见问题：

- 设定会无限膨胀，最后脱离剧情需求
- 角色弧光容易被推迟到正文阶段，导致后续大量返工

综合常见的小说创作建议，这个项目采用一条更稳的路径：

1. 先明确题材承诺和故事引擎
2. 再建立能支撑冲突的世界规则
3. 再明确角色功能与弧光
4. 然后进入长线与分卷大纲
5. 最后才开始正文生产

这条路径不是否定设定，而是要求设定必须为冲突、人物和长期推进服务。

## 推荐生产路径

### 阶段 0：外部参考与研究

位置：

- `00-reference/`

目标：

- 整理需要借鉴的作品、理论与材料
- 明确每份参考材料服务的具体问题
- 将有价值的稳定结论回写到正式层

最低产物：

- `00-reference/research-notes.md`

出口标准：

- 知道为什么读某份材料
- 知道读完后准备回写到哪一层
- 参考材料不堆成独立知识库

### 阶段 1：立项与题材承诺

位置：

- `01-foundations/`

目标：

- 明确这本书的核心卖点
- 明确读者承诺
- 明确主题问题
- 明确研究边界

最低产物：

- `project-brief.md`
- `genre-promise.md`
- `thematic-core.md`

出口标准：

- 能用几句话解释“这本书是什么”
- 能说明前期爽点和长线支撑
- 能判断哪些设定该写、哪些设定暂时不写

### 阶段 2：世界规则与设定骨架

位置：

- `02-setting/`

目标：

- 建立世界观的运行逻辑
- 建立修行、代价、资源、秩序、因果的规则
- 先做能进入剧情的必要设定，不做空转百科

最低产物：

- `02-setting/01-constitution/world-core.md`
- `02-setting/01-constitution/rule-ledger.md`
- `02-setting/02-cosmos-and-principles/README.md`
- `02-setting/03-cultivation-and-arts/README.md`
- `02-setting/04-world-layout-and-history/history-and-timeline.md`
- `02-setting/05-factions-and-polities/README.md`

出口标准：

- 关键冲突能落在世界规则里，而不是靠作者临场特判
- 境界、能力、代价和稀缺资源之间没有明显自相矛盾
- 已有规则能解释主角为何能行动、受阻、成长和失去

### 阶段 3：角色系统与弧光设计

位置：

- `03-characters/`

目标：

- 定义主角、对手、盟友、镜像角色的功能
- 明确角色关系如何推动主线
- 明确角色弧光，而不是只写静态人设

最低产物：

- `03-characters/01-indexes/role-index.md`
- `03-characters/02-protagonist/protagonist-profile.md`
- `03-characters/01-indexes/major-arc-index.md`
- `03-characters/01-indexes/relationship-map.md`

出口标准：

- 主角的欲望、缺陷、代价和变化方向清晰
- 核心角色之间的关系张力能持续推动剧情
- 角色变化不是靠结尾突然说明，而是能拆进故事阶段

### 阶段 4：长线结构与分卷大纲

位置：

- `04-plot/`

目标：

- 搭建长线冲突
- 拆出卷纲、章纲和伏笔回收路径

最低产物：

- `long-arc-blueprint.md`
- `volume-roadmap.md`
- `chapter-map.md`
- `foreshadow-payoff-ledger.md`

出口标准：

- 大纲中的高潮、反转、代价和升级能被前述规则支撑
- 主要人物弧光已经与分卷事件绑定
- 伏笔不是只记录“要埋”，而是知道何时兑现

### 阶段 5：正文草稿

位置：

- `05-drafts/`

目标：

- 在边界稳定的前提下写出可迭代的章节草稿

开写前检查：

- 规则账本没有重大空洞
- 主角与核心角色弧光已拆到卷级
- 章纲已能说明每章推进了什么

章节级检查：

- 本章推进的是哪条冲突
- 本章改变了什么关系、信息或局势
- 本章是否消耗或改写了某条既有规则
- 本章是否留下可追踪的后果

### 阶段 6：修订与连续性审计

位置：

- `06-revision/`

顺序建议：

1. 结构修订
2. 角色弧光修订
3. 设定连续性审计
4. 节奏与语言修订

说明：

- 先改结构问题，再改句子
- 先确认角色行为和世界规则自洽，再做细部润色

### 阶段 7：对外版本整理

位置：

- `07-release/`

目标：

- 整理简介、对外文案、稳定版本和发布材料

## 本项目特别强调的几条纪律

### 1. 设定必须可追责

一旦某条规则进入正文，就必须能回溯到：

- 哪份设定文档定义了它
- 它带来了什么代价
- 它会影响哪些后续剧情

### 2. 角色弧光必须前置

不能把角色成长留到“写到那时再感觉一下”。

至少要提前写清：

- 初始状态
- 核心欲望
- 核心缺陷或误判
- 关键关系压力
- 变化节点
- 阶段性代价

### 3. 重大方向问题须先确认用户

若某项判断会直接影响本文的整体基调、主题重心、核心审美、世界根基或主线方向，
则不能把“合理猜测”直接当成既定事实继续推进。

这类问题应先明确：

- 当前是否已得到用户确认
- 现有表述是稳定结论，还是暂定方案
- 若只是暂定，后续是否仍允许提出替代意见

不能因为局部推进顺手，
就绕过用户确认去替整本书定调。

## 回退与恢复纪律

### 1. 回退默认优先使用 git

凡是“回退”“撤销上一步”“恢复之前版本”这类请求，  
若目标改动可由 git 精确表达，应优先使用 git 方式完成，  
而不是优先手工重写文件。

### 2. 回退前必须先检查边界

执行回退前，必须先检查：

- `git status`
- 目标文件 diff

其目的不是走流程，  
而是确认回退边界，避免误伤无关改动。

### 3. 必须遵循最小作用原则

回退时应优先按以下顺序选择方法：

1. 只回退指定文件中的指定改动
2. 只回退指定文件到某个明确版本
3. 只回退明确指定的某次提交

除非用户明确要求，  
禁止使用 `git reset --hard`、全局 `git checkout -- .`、全局 `git restore` 等会影响无关改动的破坏性方式。

### 4. 用户改动与助手改动必须区别处理

若待回退内容是助手自己刚写入、且通过 git 可精确识别的改动，  
可静默执行最小范围回退。

若回退将涉及用户手动修改的内容，  
必须先通过 `git diff`、文件 diff 或其他可核查方式明确告知风险，  
并获得用户明确授权后才能继续。

### 5. git 无法精确表达时，再退回精确 patch

若 git 不能精确表达回退边界，  
或 git 回退会误伤无关改动，  
则应退回到最小范围的精确 patch 方式，  
而不是为了“必须用 git”而扩大回退范围。

### 6. 回退后要做最小必要校验

回退完成后，应进行最小必要校验，确认：

- 目标改动确已撤销
- 未波及无关文件

## 人物文档补充纪律

人物文档必须显式区分两层内容：

### 1. 基础设定

基础设定回答的是：

- 这个人原本是谁
- 出身、位置、性格、欲望、缺陷、能力结构是什么
- 在剧情开始前，他或她站在什么地方

这一层属于相对稳定的信息。

除非设定本身被推翻，否则不应在后续写作中反复改口。

### 2. 剧情后变化

剧情后变化回答的是：

- 经历某个阶段或节点后，这个人发生了什么位移
- 关系如何改变
- 判断如何变重、变偏或变硬
- 哪些东西已经不再和开局一样

这一层必须随着剧情推进持续回填。

否则人物就会只活在章纲里，不活在角色系统里。

## `04-plot/` 与 `05-drafts/` 的边界

这一条必须严格执行。

### `04-plot/` 只处理结构

这里放的是：

- 长线蓝图
- 卷纲
- 章纲
- 逐章功能表
- 正文细纲
- 伏笔账本
- 骨架复核

换句话说，`04-plot/` 只回答：

- 这一章为什么存在
- 这一场戏推进了什么
- 信息、关系、局势和后果如何排布

### `05-drafts/` 才处理正文

这里放的是：

- 章节草稿
- 场景草稿
- 试写片段
- 连续正文文本

换句话说，`05-drafts/` 才回答：

- 这一章具体怎么写
- 句子怎么落
- 场景怎么呈现
- 人物怎么说话、行动与感受

### 禁止串层

以下内容不应放进 `04-plot/`：

- 成段正文
- 大段描写
- 可直接连起来阅读的小说文本

以下内容不应放进 `05-drafts/`：

- 纯结构说明
- 仅供作者看的功能备注
- 章节作用清单

若某份文档主要是“给作者自己看功能”，它应放在 `04-plot/`。

若某份文档主要是“给读者阅读文本”，它应放在 `05-drafts/`。

## 文档索引工作流

《观化》的文档结构不应靠横向堆积入口来维持。

更稳的方式是：

- 树型分层
- 逐层展开
- 唯一出处

### 1. 新增目录时

若新增一个目录层级，应优先补：

- 本层 `README.md`

这个 README 至少应回答：

- 这一层整体容纳什么内容
- 阅读者下一步应去哪里

### 2. 新增正式内容时

应先判断：

- 它的唯一权威落点在哪里

确定之后：

- 只在那个位置写正文内容
- 其他地方只做引用和索引

### 3. 新增索引时

索引文档应优先采用：

- 先指目录
- 再指该目录的 `README.md`
- 由下一层 README 再继续展开

避免在高层索引里直接平铺太多深层文件。

## 参考依据

这条流程综合吸收了几类常见建议：

- 先明确 premise / genre promise，再扩写
- 世界观应围绕冲突与信念系统建设，而非只堆设定
- 大纲应与角色弧光和伏笔回收绑定
- 修订应先处理结构问题，再处理语言层问题
