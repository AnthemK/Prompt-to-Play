# Document Governance

## Audience

面向维护《观化》文档边界的任何协作者。

## 顶层结构原则

《观化》顶层结构分为两类：

### 1. 正式层

包括：

- `docs/`
- `00-reference/`
- `01-foundation/`
- `02-setting/`
- `03-characters/`
- `04-plot/`
- `05-drafts/`
- `06-revision/`
- `07-release/`

正式层必须满足：

- 可长期维护
- 遵守唯一出处
- 可被新人逐层阅读

### 2. 运行层

包括：

- `08-experimental-writing/`
- `09-cache/`

运行层只服务当前工作。

运行层允许：

- 临时性
- 任务导向
- 生命周期管理

运行层不应成为新的正式知识库。

## 根目录文件边界

### `README.md`

只负责：

- 项目入口
- 当前流程概览
- 从哪里开始读

不负责：

- 具体设定细节
- 角色档案
- 章节级大纲

### `MEMORY.md`

只负责：

- 当前状态
- 已确定原则
- 当前风险
- 接手顺序

不负责：

- 完整创作历程
- 灵感流水账
- 长期设定正文

### `TODO.md`

只负责：

- 当前优先级
- 当前不做什么
- 阶段切换条件

不负责：

- 详细设定
- 完整剧情大纲

## 目录边界

### `00-reference/`

放外部参考与研究笔记。

适合放：

- 名作参考
- 理论参考
- 研究问题
- 读后提炼

这里不直接承担正式设定、正式角色或正式大纲职能。

### `docs/`

放稳定规则与治理文档。

适合放：

- 流程规范
- 文档边界
- 质量标准
- 协作与执行规则

这里不负责小说本体的题材承诺、主题问题与审美原则。

### `01-foundation/`

放小说本体层的立项资料。

适合放：

- 项目定位
- 卖点与承诺
- 主题问题
- 前期研究边界
- 小说本体想成为什么样的作品

不适合放：

- cache 规则
- 目录治理
- 命名规范
- 任务执行规则

补充：

- `01-foundation/` 负责小说架构与本体定位
- `docs/` 负责协作规则与执行规则

### `02-setting/`

放世界观和设定骨架。

适合放：

- 宇宙观
- 修行体系
- 势力结构
- 历史时间线
- 规则账本

当前建议分层：

- `01-foundations/`
  - 世界底层规则
- `02-world-order/`
  - 天下结构与跨势力秩序
- `03-taiyi/`
  - 主宗门专章

### `03-characters/`

放角色资料与弧光设计。

适合放：

- 主角资料
- 主要角色功能
- 弧光分解
- 关系网络

### `04-plot/`

放大纲和结构材料。

适合放：

- 长线蓝图
- 卷纲
- 章纲
- 伏笔回收表

### `05-drafts/`

只放正文草稿。

不要把设定修订记录和审校意见混进来。

## 占位符约定

在设定与大纲阶段，如果主角、地点或某些专名尚未最终定稿，可以临时使用占位符。

推荐格式：

- `{{MC_NAME}}`
- `{{MAIN_COUNTY}}`
- `{{RIVAL_NAME}}`

使用规则：

- 只建议用于 `01-foundation/`、`02-setting/`、`03-characters/`、`04-plot/`
- 进入 `05-drafts/` 前应尽量替换为正式名称
- 正文草稿阶段不建议继续保留大量占位符，否则语言会失真

### `06-revision/`

放修订和问题清单。

适合放：

- 连续性问题
- 节奏问题
- 重写计划
- 审校反馈

### `07-release/`

放对外整理材料。

## 文档迁移原则

- 还在探索中的内容，优先放阶段目录
- 已经确认稳定的规则，再收束到稳定文档
- 一条信息只应有一个“最终可信版本”
- 文档应保持分层、分级与目录自洽，不应长期平铺堆叠
- 每新增一个子层目录，原则上应补对应 `README.md` 或入口索引
- 重要文档新增后，应同步检查上层索引、相关目录索引与交叉引用
- 如果一份文档已经承担两个层级的职能，应优先拆分，而不是继续加内容

## 树型索引原则

《观化》的文档结构应尽量保持为：

- 树型分层
- 逐层展开
- 唯一出处

此外，索引层职责固定分为四类：

### `README`

目录入口，回答“这一层是什么”。

### `index`

人工阅读索引，回答“这一类内容整体怎么理解”。

### `meta`

结构化导航索引，回答“去哪里找”和“哪些文件直接相关”。

### `cache`

当前任务的临时上下文包，回答“这次工作最少必须知道什么”。

新文档若承担索引职责，应优先判断自己属于哪一类，而不是混写。

### 1. README 的职责

每一层目录下的 `README.md` 应同时完成两件事：

- 概括这一层目录整体在解决什么问题
- 指向下一层应该去哪里继续读

也就是说，README 不是只做极薄的路标。

它应对“这一层整体容纳什么内容”给出足够清楚的概括。

但它不应越级承担更深层正文。

### 2. 只展开到下一层

若一个目录下面还有子目录，则本层 `README.md` 原则上只应：

- 解释本层作用
- 列出下一层子目录或入口文件
- 告诉阅读者先去下一层 `README.md`

不应在本层 README 中继续大段展开下下层内容。

### 3. 唯一出处

同一项具体内容，应只有一个权威落点。

其他文档只负责：

- 引用它
- 指路到它

不重复复述，不平行再写一版。

### 4. 引用优先于复述

若某处只是需要提醒读者“去哪看”，应优先写：

- 这一问题请去哪个目录
- 进入后先读哪个 README
- 若需要再看哪份正式文件

而不是在当前位置重新总结一遍正文内容。

## 特别提醒

《观化》高度重视世界规则和角色弧光，因此最容易出现两类污染：

1. 设定文件开始偷偷承担剧情大纲功能
2. 角色文件只写静态人设，不写故事功能和变化

发现这两种情况时，应优先拆文档，而不是继续往同一份文件里补内容。
