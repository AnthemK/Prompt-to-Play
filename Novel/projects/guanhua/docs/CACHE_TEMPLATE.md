# Cache Template

## 用途

这是一份《观化》项目的标准 cache 模板。

它用于：

- 为某个具体任务临时聚合最小必要信息
- 降低反复穿梭树型文档结构的成本
- 提供可销毁、可重建的任务包

它不用于：

- 替代正式设定
- 替代正式角色文档
- 替代正式大纲
- 替代正式正文草稿

## 文件命名建议

统一使用：

- `<task>-pack.md`

例如：

- `volume-01-opening-pack.md`
- `volume-01-big-match-pack.md`
- `experimental-writing-opening-style-pack.md`

补充要求：

- 全小写
- 单词之间用 `-` 连接
- 不使用空格
- 不使用中文文件名
- 不在名称里堆叠 `final`、`new`、`temp` 之类无语义词

## 放置建议

建议放在：

- `09-cache/active/`

任务结束后，默认应删除。

若确有短期保留必要，可临时移入：

- `09-cache/archive/`

但 `archive/` 应尽量少用。

# Cache Pack: <name>

## 用途

这份 cache 服务于什么任务。

写法要求：

- 只写当前任务
- 不写泛泛而谈的长期目标

示例：

- 这份 cache 用于卷一开篇写作阶段，服务第一章到第三章的正式落文与实验性改写对照。

## 生命周期

明确说明：

- 何时创建
- 何时结束
- 到期后应删除还是可短暂归档

示例：

- 创建时机：
  - 卷一开篇进入正式落文前
- 生命周期：
  - 仅覆盖开篇写作与第一轮文气校准周期
- 结束条件：
  - 第一章到第三章完成第一轮正式草稿并校准文气后失效
- 失效后处理：
  - 默认删除，不长期保留

## 当前任务目标

把这次任务真正要完成的事情写清。

建议控制在 3 到 5 条以内。

示例：

- 完成第一章正式草稿
- 校准开篇文气
- 确保旧院、晨点、主角位置感成立

## 必须遵守的硬约束

这里只放当前任务最关键的硬约束。

不要把全项目所有规则都再抄一遍。

建议控制在 5 到 10 条以内。

示例：

- 不得写成仙气飘然的入门篇
- 第一章只允许写到晨点前，不推进到正式点验流程
- 文风必须“冷而不硬，实而不俗，雅而不浮，克制里带锋芒”
- 不得用设定说明替代现场感

## 最小必要上下文

这里只写当前任务真的必须知道的稳定结论。

每一条后面都必须附正式来源。

推荐格式：

### <编号>.<简短标题>

> <一句话稳定结论>

来源：
`<canonical-file-path>`

示例：

### 1. 主角开篇定位

> 陆守一当前是十四岁的太一门外门弟子，练气三层，天赋方向为“内秀于心”。

来源：
`03-characters/02-protagonist/protagonist-profile.md`

## 推荐阅读顺序

这里只列当前任务最值得读的正式来源，不必多。

建议控制在 3 到 8 个入口。

推荐格式：

1. 先读：
   - `<file>`
   - 作用：<一句话>
2. 再读：
   - `<file>`
   - 作用：<一句话>

## 正式来源

这里列这份 cache 依赖的全部正式文件。

要求：

- 只列路径
- 不复述正文

## 本轮若有价值修改，应回写到哪里

这一节必须明确。

要求写清：

- 哪类变化该回写
- 回写到哪个正式文件

示例：

- 若主角开篇心理重心发生稳定调整，回写：
  - `03-characters/02-protagonist/protagonist-profile.md`
- 若第一章场景顺序发生稳定调整，回写：
  - `04-plot/volume-01/chapter-001-010-detail-outline.md`
- 若正式正文措辞被采纳，回写：
  - `05-drafts/volume-01/chapter-001.md`

## 失效条件

必须明确哪些正式文件一旦变化，这份 cache 就需要重建。

示例：

- 若 `03-characters/02-protagonist/protagonist-profile.md` 发生主角开篇定位调整，则本 cache 失效
- 若 `04-plot/volume-01/chapter-001-010-detail-outline.md` 发生场景顺序调整，则本 cache 失效
- 若开篇文风要求发生修订，则本 cache 失效

## 当前状态

建议只用非常短的状态标记，例如：

- `active`
- `stale`
- `done`

并加一句说明当前处于哪个阶段。

不建议使用更多自造状态词。

## 清理说明

任务结束后应执行：

1. 先将当前状态改为 `done` 或 `stale`
2. 确认有价值修改已经回写正式文件
3. 确认正式文件已成为唯一可信版本
4. 删除本 cache

若确有必要短期保留，才允许移入 `archive/`。

若移入 `archive/`，也应尽快在下一轮相关任务前完成删除。
