# Cache Policy

## Audience

面向维护《观化》工作缓存机制的任何协作者。

这份文档不定义具体缓存脚本实现。

它只定义：

- cache 是什么
- cache 不是什么
- cache 何时创建、何时销毁
- cache 与正式文档的关系
- cache 的更新与失效原则

## 一、核心定义

《观化》的 cache，不是第二套知识库，也不是另一份设定集。

它的唯一目的，是：

- 为某个当下任务临时聚合最小必要信息
- 减少反复穿梭整棵树形文档结构的成本
- 提升协作者在某个工作周期内的效率

更准确地说，cache 是：

- 面向任务的
- 临时的
- 可销毁的
- 派生的

## 二、cache 不是什么

cache 不是：

- 正式设定文档
- 正式角色文档
- 正式卷纲或章纲
- 正式正文草稿
- 正式修订记录

cache 也不是：

- 长期保留的第二摘要层
- 与正式文档并列的另一份可信版本
- 允许自由生长的临时笔记堆

如果 cache 开始承担上述角色，它就已经越界。

## 三、唯一出处原则

《观化》继续坚持：

- 同一项具体内容，只有一个权威出处

cache 只能：

- 摘要
- 聚合
- 引用
- 指路

不能成为新的权威来源。

若 cache 中出现了有价值的新判断、新修正、新措辞或新结构，处理方式应当是：

1. 判断它的正式归属位置
2. 回写到对应原始文件
3. 让 cache 继续引用该原始文件

而不是长期把这部分内容留在 cache 里。

## 四、生命周期原则

cache 必须有明确生命周期。

### 1. 创建条件

只有在以下情况之一成立时，才应建立 cache：

- 某一阶段任务需要反复调用多份正式文档
- 某一位协作者的任务范围足够集中，适合打包最小必要上下文
- 若不建 cache，会频繁重复读取正式树形结构，明显浪费成本

### 2. 存续条件

cache 只在对应任务周期内存续。

例如：

- 卷一开篇写作周期
- 卷一大比写作周期
- 某次实验性写作周期
- 某轮连续性审计周期

### 3. 销毁条件

当以下任一情况成立时，cache 应视为到期并销毁：

- 该任务已经完成
- 该任务方向已经改变
- 其所依赖的正式文件发生重大更新，导致该 cache 大量失效
- 其继续保留的维护成本已经高于再次生成的成本

## 四点五、命名规则

cache 文件命名必须统一、短而可判读。

### 1. 基本格式

统一使用：

- `<task>-pack.md`

其中：

- `<task>` 必须描述任务域
- `pack` 固定保留，用来明确这是任务包，不是正式正文或正式设定

### 2. 推荐命名结构

建议按：

- `<scope>-<focus>-pack.md`

例如：

- `volume-01-opening-pack.md`
- `volume-01-big-match-pack.md`
- `experimental-writing-opening-style-pack.md`
- `continuity-audit-volume-01-pack.md`

### 3. 命名内容要求

命名应满足：

- 全小写
- 单词之间用 `-` 连接
- 不使用空格
- 不使用中文文件名
- 不包含日期，除非同一任务同名冲突且必须区分批次

### 4. 命名长度要求

命名应尽量控制在：

- 3 到 6 个词

过短会失去辨识度，过长会失去可读性。

### 5. 不推荐的命名

避免：

- `cache1.md`
- `temp-pack.md`
- `new-version-pack.md`
- `opening-final-final-pack.md`

这些名字都缺少任务语义，后续难以维护。

## 五、回写原则

cache 中若产生真正有价值的内容，不应直接把 cache 留作“以后再说”。

更稳的流程应是：

1. 从 cache 中识别有价值变化
2. 确认它属于哪个正式文件
3. 回写到唯一权威出处
4. 必要时刷新或重建 cache

这里最重要的是：

- 价值留在正式树上
- 缓存只保留任务便利性

## 六、版本与可信度原则

正式文档与 cache 的可信度永远不应并列。

可信度顺序必须固定为：

1. 正式原始文件
2. 由原始文件派生出的 cache

若 cache 与正式文件冲突：

- 一律以正式文件为准

若发现 cache 更合理：

- 不应直接改 cache 结束
- 而应回写正式文件，再刷新 cache

## 七、结构原则

cache 的结构应尽量轻。

推荐只保留：

- 任务目标
- 最小必要摘要
- 来源引用
- 下一步动作
- 失效条件

不推荐在 cache 中塞入：

- 大段重复设定正文
- 大量完整角色档复制
- 整卷大纲全文复制
- 无边界增长的灵感材料

## 七点五、来源格式原则

cache 中所有“来源”字段，必须使用：

- 项目内相对路径

例如：

- `03-characters/02-protagonist/protagonist-profile.md`
- `04-plot/volume-01/chapter-outline.md`

明确禁止：

- 绝对文件系统路径
- 暴露本机用户名、机器目录或工作区私有结构的信息
- 依赖单机环境才能成立的硬编码链接

cache 中的来源字段，其职责只是：

- 指明唯一权威出处在哪里

而不是承担跨环境可点击跳转能力。

如果未来需要技术化跳转，应由独立工具层解决，而不是把绝对路径写进 cache 正文。

## 八、与树型文档结构的关系

《观化》的长期主结构仍然是：

- 树型分层
- 逐层展开
- 唯一出处

cache 只是建立在这棵树上的临时任务包。

因此：

- 树负责长期正确性
- cache 负责短期效率

cache 不能反过来取代树。

## 九、Git 与持久化原则

cache 不应进入正式版本控制。

明确要求如下：

### 1. 不纳入 Git 备份

cache 目录或 cache 文件应被 `.gitignore` 忽略。

### 2. 不应提交

无论内容多有用，cache 本身都不应直接提交进仓库历史。

真正有价值的变化，应回写到正式文档后提交正式文档。

### 3. 可重建优先于可保存

cache 的设计哲学应是：

- 低成本重建
- 而不是长期保存

## 九点五、目录使用规则

cache 当前允许的目录结构为：

- `09-cache/active/`
- `09-cache/archive/`

### 1. `active/`

这里只放当前仍在生命周期内的 cache。

进入 `active/` 的文件必须满足：

- 任务正在进行
- 内容仍可能被继续使用
- 尚未完成清理动作

### 2. `archive/`

`archive/` 不是常规目录。

它只允许在以下情况使用：

- 任务刚结束，但短期内仍需对照
- 某次讨论、验收或回溯尚未完成
- 计划在极短周期内直接删除，但需要一个过渡位置

默认原则：

- 能删就删
- 不要把 `archive/` 用成永久垃圾堆

## 十、更新机制原则

cache 的更新不应主要依赖人工反复阅读和手工重写。

更理想的方向是：

- 从正式文档中自动抽取最小必要信息
- 按任务模板自动生成或半自动生成 cache

这意味着未来若实现技术方案，应优先考虑：

- 结构化元数据
- 规范标题抽取
- 脚本生成
- 明确失效标记

而不是依赖协作者频繁消耗额外成本人工维护。

## 十点五、状态值规则

cache 的 `当前状态` 字段建议只允许以下三种值：

- `active`
- `stale`
- `done`

含义固定为：

### 1. `active`

- 当前任务仍在进行
- cache 仍可继续使用

### 2. `stale`

- 正式来源已变化
- 这份 cache 暂不可继续信任
- 必须重建或直接删除

### 3. `done`

- 任务已完成
- 相关价值修改应已回写
- 下一步应删除，或极短期移入 `archive/`

禁止自造大量状态词，例如：

- `almost-done`
- `v2`
- `keep-for-now`

这些词会让清理规则失控。

## 十一点五、清理规则

cache 的清理不应依赖“以后有空再说”。

应执行一个固定清理流程。

### 1. 清理前检查

在清理某份 cache 前，必须确认：

- 其中有价值修改是否已经识别
- 这些修改是否已经回写到正式文件
- 正式文件是否已成为唯一可信版本

### 2. 清理动作顺序

建议顺序固定为：

1. 将 `当前状态` 改为 `done` 或 `stale`
2. 完成必要回写
3. 重新检查正式来源是否足够承接后续工作
4. 直接删除该 cache

### 3. 允许进入 `archive/` 的例外情况

只有在以下情况，才允许从 `active/` 移入 `archive/`：

- 任务虽结束，但同一轮讨论尚未彻底关闭
- 短期内仍需人工对照
- 删除动作明确延后，但不会无限拖延

### 4. `archive/` 的清理要求

进入 `archive/` 的 cache 也不是永久保留。

建议：

- 在下一轮相关任务开始前，优先检查并删除过期 archive
- 若 archive 中内容已全部回写，直接删除
- 若 archive 中内容仍有价值但未回写，先回写，再删除

### 5. 失效优先于保留

若一份 cache 已经明显失效，而是否保留又拿不准，应优先：

- 标记为 `stale`
- 尽快删除

不要让过期 cache 在目录中长期悬挂。

## 十一、当前建议

在《观化》项目里，cache 若要正式落地，建议按以下顺序推进：

1. 先定义 cache 放在哪一层
2. 再定义 cache 命名规则
3. 再定义哪些任务值得建 cache
4. 再定义最小字段模板
5. 最后再考虑脚本化生成

不要在规则未定前先大量创建 cache。
