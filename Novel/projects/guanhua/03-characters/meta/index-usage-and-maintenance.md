# Character Index Usage And Maintenance

## 用途

这份文档说明 `character-index.json` 应该怎么用、什么时候改、改到什么程度为止。

它处理的是维护规则，不替代角色正式设定。

## 一、先读什么

如果你只是想快速找到某个角色的正式 Markdown：

1. 先读 `character-index.json`
2. 找到该角色的 `canonical_file`
3. 若 `file_mode = shared`，再根据 `canonical_anchor` 跳到对应小节

如果你要理解人物本身，不要停在 JSON。

继续去读：

- `canonical_file`
- 必要时联读 `plot_refs` 指向的剧情文件

## 二、这个索引能回答什么

它只应该回答机械问题，例如：

- 某个角色现在去哪个 Markdown 文件看
- 该角色是独立立传还是合并收录
- 该角色当前和哪些剧情文件直接相关
- 该角色在当前卷第一次出现于哪个阶段

它不应该承担：

- 人物弧光分析
- 情感状态说明
- 大段性格描述
- 剧情解释

这些内容只能留在正式 Markdown 文档中。

## 三、什么时候需要改索引

出现以下情况时，必须检查并更新 `character-index.json`：

1. 新角色第一次进入正式文档
2. 某个角色从合并文件拆出，变成独立立传
3. 某个角色的正式归档文件发生迁移
4. 某个角色在当前卷的相关剧情文件发生稳定变化
5. 某个角色新增了稳定阵营归属，或发生了阵营位移

补充：

如果只是角色正文措辞调整、弧光加深、心理更细，不必因此修改索引，除非这些变化影响了文件位置、阵营列表或剧情连接。

## 四、什么时候应该先改 Markdown，再改索引

原则上永远是：

先改正式 Markdown

再改 `character-index.json`

因为 JSON 不是权威设定源，它只是导航层。

正确顺序应为：

1. 先把角色正式段落补到对应 Markdown
2. 确认该角色的正式归属文件已经稳定
3. 再更新 `canonical_file`、`canonical_anchor`、`plot_refs` 等机械字段

禁止反过来先改索引，再让正式人物档长期缺位。

## 五、如何判断角色该放单独文件还是合并文件

遵守当前项目规则：

- 主要角色必须单独立传
- 次要角色、同类角色可以合并到群像文件

更具体地说：

适合 `single` 的情况：

- 对主线有长期结构性影响
- 需要单独承载角色弧光
- 后续会被反复单独调用

适合 `shared` 的情况：

- 当前仍主要承担卷级或阶段性功能位
- 作为群像的一部分更合理
- 暂时不需要独立展开完整人物弧光

一旦角色升格，必须做两件事：

1. 先把正式人物内容拆到独立 Markdown
2. 再把索引中的 `file_mode` 从 `shared` 改为 `single`

## 六、字段维护规则

`id`

稳定使用，不随称呼细调而频繁修改。

`name`

使用当前正式中文名。

`factions`

使用列表，不使用单值。这样当角色发生阵营位移、兼具多重归属时，不需要改字段结构。

`canonical_file`

必须始终指向当前唯一权威 Markdown。

`canonical_anchor`

只在 `shared` 模式下填写，并且必须与文件中的真实标题完全一致。

`plot_refs`

只放文件级弱连接，不复制剧情内容。

## 七、维护时的最小检查清单

每次修改索引后，至少检查：

1. 新增或修改的角色是否已有正式 Markdown 段落
2. `canonical_file` 是否真实存在
3. 若为 `shared`，`canonical_anchor` 是否能在文件中找到
4. `plot_refs` 是否仍指向当前有效的剧情文件
5. 是否把文学内容误写进了 JSON

## 八、给人和后续协作者的工作建议

如果任务偏检索：

先用 `character-index.json` 缩小范围，再去读少量正式 Markdown。

如果任务偏创作：

把 JSON 当成入口，不要把 JSON 当成可直接写作的人物设定。

如果任务偏维护：

先修正式文档，再回收索引，不要让 JSON 和 Markdown 互相打架。
